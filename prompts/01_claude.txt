  ---
  Architecture Decisions

  Stack I'd recommend:
  - Backend: Node.js + Express + SQLite (via Drizzle ORM) — lightweight,
  file-backed, schema-defined, zero infra overhead
  - Frontend: React + Vite + TanStack Query + React Router
  - Auth: Simple JWT (hardcoded credentials are fine for this scope — the key is
   demonstrating you know the security implications)
  - AI: Anthropic Claude API (you're already using Claude Code, signals
  familiarity)

  Why SQLite over Postgres? The challenge says "database or file-backed store."
  SQLite gives a real schema, real foreign keys (cascade deletes), real SQL —
  but no Docker dependency for the DB. This keeps setup simple while satisfying
  the constraint. It's also an honest trade-off to document in the README.

  ---
  Key Design Considerations

  Cascade delete should be enforced at the DB level (FK constraint), not just in
   application code. This is worth highlighting — it shows you understand where
  constraints belong.

  POST /sessions/search is an unusual REST choice (search via POST). The spec
  explicitly asks for it. I'd implement it but mention in the README that a GET 
  /sessions?filter=... pattern would be more idiomatic.

  Local Risk Score — this is a chance to show product thinking. I'd design rules
   that are meaningful:
  - +15 — country not in a "safe" allowlist
  - +25 — LOGIN_ATTEMPT followed by FORM_SUBMIT within 5s
  - +20 — FORM_SUBMIT with sensitive metadata.formFields (card_number, cvv,
  password)
  - +15 — multiple LOGIN_ATTEMPT events (brute force signal)
  - +10 — session status is already SUSPICIOUS/DANGEROUS
  - +15 — very short durationMs on form submissions (bot-like speed)

  Capped at 100. Computed on the backend, stored or derived on read.

  ---
  Security Surface (for README)

  At least 2 required, I'd document 4:
  1. JWT secret — must come from env, not hardcoded; short expiry
  2. CORS — restrict to frontend origin, not *
  3. Input validation — sanitize all POST/PUT bodies (Zod on the backend)
  4. Rate limiting on AI endpoint — the LLM call is expensive and abusable; add
  a simple rate limiter middleware

  ---
  Bonus Features

  Both are worth doing:
  - AI Risk Summary — straightforward with Anthropic SDK; the prompt engineering
   is where you show judgment
  - Docker Compose — simple 2-service compose (backend + frontend served via
  nginx or Vite preview); makes the reviewer's life easy and signals production
  awareness

  The spec explicitly says the README is evaluated as a design document. I'd
  structure it as:
  1. Quick start
  2. Architecture & trade-offs (why SQLite, why this stack, cascade delete

  ---
  Revised Architecture

  Backend:
  - Java 25 + Spring Boot 3.3.6
  - Spring Web (REST controllers)
  - Spring Data JPA + Hibernate (ORM)
  - Spring Security + JJWT (JWT auth)
  - Spring AI (Anthropic integration — native Spring, clean abstraction for the
  bonus)
  - Flyway (schema migrations — gives auditable, versioned DDL)
  - Jakarta Bean Validation (@Valid on request bodies)
  - Lombok (reduce boilerplate)
  - PostgreSQL driver

  Frontend: React + Vite + TanStack Query + React Router + Tailwind

  ---
  Key Rethinks

  PostgreSQL changes the Docker situation. Since Postgres can't just be a file,
  Docker Compose becomes essentially mandatory even for local dev — which means
  the "bonus" containerization task is now foundational. I'd structure it from
  day one: docker-compose.yml with 3 services: db, backend, frontend.

  Flyway for migrations is the right call here. Each schema change is a
  versioned SQL file (V1__init.sql, V2__add_indexes.sql). This is production
  thinking and worth highlighting in the README.

  Cascade delete — two layers:
  - PostgreSQL FK: REFERENCES sessions(id) ON DELETE CASCADE (DB enforces it
  regardless of how data is deleted)
  - JPA: @OneToMany(cascade = CascadeType.ALL, orphanRemoval = true) on the
  Session entity
  - Document this deliberately: DB constraint is the safety net; JPA is the ORM
  convenience

  Layered architecture:
  Controller (DTOs) → Service (business logic + risk score) → Repository (JPA)
  Entities never leave the service layer — DTOs are the API surface. This
  matters for security (no accidental field leakage).

  Risk score — computed in a RiskScoringService, called when building the
  response DTO. Not persisted. This keeps it always fresh and avoids a
  stale-data problem if events are added later.

  ---
  Spring AI for the Bonus

  Spring AI 1.x has native Anthropic support. This is cleaner than using the raw
   SDK:
  @Autowired ChatClient chatClient; // Spring AI abstraction
  One trade-off to document: Spring AI abstracts the provider, so swapping to
  OpenAI later is trivial — but it adds a framework dependency. Worth noting.

  ---
  Security Rethink for Spring

  1. JWT — SecurityFilterChain with a JwtAuthenticationFilter; secret from
  application.yml environment variable, never hardcoded
  2. CORS — CorsConfigurationSource bean, restricted to frontend origin
  3. Input validation — @Valid + Jakarta constraints on all request DTOs;
  @ControllerAdvice for unified error responses (no stack traces leaking to
  clients)
  4. Rate limiting on AI endpoint — Bucket4j is the standard Spring-compatible
  choice; annotate the AI endpoint with a rate limit interceptor

  ---
  Liquibase — Production-Grade Setup

  With Liquibase, the schema lifecycle is fully controlled by changesets:
  - db/changelog/db.changelog-master.xml as the root
  - V001__create_sessions.sql, V002__create_events.sql as individual changesets
  - spring.jpa.hibernate.ddl-auto=validate — Hibernate only validates the schema
   matches the entities, never touches DDL
  - spring.liquibase.enabled=true — Liquibase runs on startup and applies
  pending changesets

  Worth documenting: this means the DB is the source of truth for schema, not
  the JPA entities. A drift between entities and schema will be caught at
  startup, not silently ignored.

  ---
  Spring Actuator — /health

  Spring Boot Actuator gives this almost for free. The considerations:
  - Expose /actuator/health — shows DB connectivity (Postgres datasource health
  indicator is auto-configured)
  - Expose /actuator/info — app version, build info
  - Critically: lock down all other actuator endpoints (/actuator/env,
  /actuator/beans, etc.) — they leak sensitive data. In application.yml: expose
  only health and info publicly, require auth for the rest or disable them
  entirely
  - This is a legitimate security point worth documenting in the README

  ---
  Acceptance Criteria

  This is important — it gives the work structure and demonstrates you think in
  testable behaviors, not just features. I'd define them at three levels:

  API-level (integration tests with Spring's MockMvc or @SpringBootTest + 
  Testcontainers for Postgres):
  - Given a valid JWT, when GET /sessions is called, then 200 and a list is
  returned
  - Given no JWT, when any protected endpoint is called, then 401
  - Given a session with events, when DELETE /sessions/:id, then session and all
   events are deleted (cascade verified by querying events after)
  - Given a search request with status=DANGEROUS, then only DANGEROUS sessions
  are returned
  - Given a session with a LOGIN_ATTEMPT followed by FORM_SUBMIT within 5s, then
   risk score ≥ 25

  Service-level (unit tests):
  - Risk scoring rules tested in isolation with mocked event lists
  - Each rule contributes the expected points
  - Score is capped at 100

  Frontend (Vitest + React Testing Library):
  - Session list renders status badges correctly
  - Delete confirmation modal appears before deletion
  - Risk score badge displays correct color tier (green/amber/red)
